Scaffy

Set of best practices, approaches & tools by Railsware client engineers to write logical, modular, easily maintainable & extendable CSS. Scaffy on Github

Achievements board for large-scale projects

  • Look identically everywhere

    Doesn't matter where your object will take place it looks the same. It is possible by creating unique classes for each object. For example .navigation in .header will look identically the same in .footer or in the .content area. That means if you need to modify navigation in footer, you have to create another type of it, like .sub-navigation.

    Selector should describe a main object, be as short as it can & don't make potential conflict: 

    .player .avatar  -> .player-avatar
.navigation li   -> .navigation-item
.lightbox .close -> .lightbox-close

  • Minimize style inherritance

    When you have a high nesting of objects inside each other the deepest one will take a blow of all styles you implied to its parents' tags. To get this achievement don't set styling to tags. Your CSS should have this look in ideal: 
    ul.navigation    -> .navigation
ul.navigation li -> .navigation-item

  • Object modifiers

    Let's say you have 3 sizes and 3 colors of buttons. In this case its better to write .button as a base and create modifiers of it keeping separate structure from skin: 
    .button { ... }
.button.tiny { ... } /* Defines shape and size */
.button.positive { ... } /* Defines color */
    Also you can move such objects to a group — a separate stylesheet where you store all modifiers of the current class.

  • Code style & conventions

    Depending on your project needs team should define a code style to be used accross the project. Here is an example: 
    .selector {
  background: #ccc url('image.png') 0 0 no-repeat;
  font: 14px 'Helvetica Neue', Helvetica, sans-serif
  padding: 0 4px 2px;
  z-index: 5;
}

.selector-child {
  display: inline-block;
  -webkit-transition: all .2s;
     -moz-transition: all .2s;
      -ms-transition: all .2s;
       -o-transition: all .2s;
          transition: all .2s;
}
  • Using SASS in full power by having shared mixins & variables across the project
  • Organize endless amount of Skins & Layouts without code duplicate

So what's inside?

Framework

  • base.sass — linker for all framework stylesheets
  • config.sass — all shared variables store here. Also, links mixins at the end
  • grids.sass — rock solid grids that allow you to build any layout no matter how complex is it. Works perfectly in IE6+ and all others
  • mixins.sass — all SASS mixins go here. No need to use monster tools with thousands of mixins, just create mixins for your needs and use them across the project.
  • reset.sass — resets everything to zero margin, padding + some black magic hacks to fix browsers inconsistencies
  • spaces.sass — use to modify the default spacing between objects
  • type.sass — stores all typography related style definitions

Layouts

  • *.sass — place for general layout styling. Store body, wrapper, content and other layout specific blocks here. Add a link to a config.sass to use shared mixins with variables. You can store all kinds of layouts here like: application.sass, homepage.sass, admin.sass

Library

  • *.sass — place for groups like buttons, text-inputs, selects, radios, checkboxes and other elements, which you can combine. Add a link to a config.sass to use shared mixins with variables. To specify multiple libraries for different layouts move groups to separate folders inside the library: /library/application/ /library/admin/

Modules

  • *.sass — use to create modules such as login-block.sass, modal.sass, votebar.sass, dropdown.sass, lightbox.sass and so on. Add a link to a config.sass to use shared mixins with variables.
  • To specify multiple modules sets for different layouts create folders inside:

    /modules/application/

    /modules/admin/

  • If module gets too big, slice it to keep code small:

    /modules/participant/info.sass

    /modules/participant/status.sass

    /modules/participant/controls.sass

    /modules/participant/states.sass
