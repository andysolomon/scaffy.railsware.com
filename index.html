Achievements board for large-scale projects

  • Look identically everywhere

    Doesn't matter where your object will take place it looks the same. It is possible by creating unique classes for each object. For example .navigation in .header will look identically the same in .footer or in the .content area. That means if you need to modify navigation in footer, you have to create another type of it, like .sub-navigation.

    Selector should describe a main object, be as short as it can & don't make potential conflict: 

    .player .avatar  -> .player-avatar
.navigation li   -> .navigation-item
.navigation li   -> .navigation > li /* In case navigation is flat and doesn't have a big nesting */
.lightbox .close -> .lightbox-close

  • Minimize style inherritance & reduce cascades

    When you have a high nesting of objects inside each other the deepest one will take a blow of all styles you implied to its parents' tags. Another tricky situation happens if you have a cascade and your object in fact location dependent. To get this achievement reduce cascades where possible & don't set styling to tags. Your CSS should have this look in ideal: 
    ul.navigation    -> .navigation
.modal-popup .login-content .buttons .login-button  -> .modal-button
.modal-popup .login-content #social_connect_buttons -> .modal-buttons

  • Object modifiers

    Let's say you have 3 sizes and 3 colors of buttons. In this case its better to write .button as a base and create modifiers of it keeping separate structure from skin: 
    .button { ... }
.button.tiny { ... } /* Defines shape and size */
.button.positive { ... } /* Defines color */
    Also you can move such objects to a group — a separate stylesheet where you store all modifiers of the current class.

  • Code style & conventions

    Depending on your project needs team should define a code style to be used accross the project. Here is an example: 
    .selector {
  background: #ccc url('image.png') 0 0 no-repeat;
  font: 14px 'Helvetica Neue', Helvetica, sans-serif
  padding: 0 4px 2px;
  z-index: 5;
}

.selector-child {
  display: inline-block;
  -webkit-transition: all .2s;
     -moz-transition: all .2s;
      -ms-transition: all .2s;
       -o-transition: all .2s;
          transition: all .2s;
}

  • Mighty SASS

    With SASS you can create shared variables config & set of mixins for CSS properties with vendor prefixes. Also it is extremely useful to control specific IE versions support by creating mixins for cross-browser compatible inline-block or opacity. Finally you save time writing less. Here is an example of a typical config: 
    // Colors
$base-color: #000
$opaque:     #333
$neutral:    #666
$pale:       #999
$bare:       #aaa
$firm:       #9bc345
$sharp:      #fff
$contrast:   #000
$positive:   #9cc346
$negative:   #c34646
$link:       #0072ff

// Typography
$base-font: Arial, Helvetica, sans-serif
$firm-font: Georgia, serif
$h1: 24px
$h2: 20px
$h3: 18px
$h4: 16px
$h5: 15px
$h6: 11px
$base-font-size: 14px

// Spaces
$space-x: 3px
$space-s: 5px
$space-m: 10px
$space-l: 20px
    Create only mixins you need. No sense to keep not used code that just confuses you. So here's mixins.sass example (to call mixin use + instead of equal): 
    =transform($properties)
  -webkit-transform: $properties
  -moz-transform:    $properties
  -ms-transform:     $properties
  -o-transform:      $properties
  transform:         $properties

=user-select($properties)
  -webkit-user-select: $properties
  -moz-user-select:    $properties
  -ms-user-select:     $properties
  user-select:         $properties

=selection($background, $color)
  &::selection
    background: $background
    color: $color

  &::-moz-selection
    background: $background
    color: $color
    Note that there are two types of mixins: private and shared. The above example describes shared mixins. If you need to automate some calculations inside specific module create private mixin right in the begining of the current module file: 
    // scrollbox.sass

=center-slider($size)
  height: $size
  margin-left: -($size / 2)
  right: -($size / 2)
  width: $size

  • Class name should live for years

    Class name should describe the function of an object. If we're talking about color appearance (object modifier) then it is useful to abstract naming cause it's appearance can change quite frequently. As an example let's take a .button: 
    .button.green -> .button.positive
.light-blue -> .pale
.white -> .contrast

So what's inside?

Framework

  • base.sass — linker for all framework stylesheets
  • config.sass — all shared variables store here. Also, links mixins at the end
  • grids.sass — rock solid grids that allow you to build any layout no matter how complex is it. Works perfectly in IE6+ and all others
  • mixins.sass — all SASS mixins go here. No need to use monster tools with thousands of mixins, just create mixins for your needs and use them across the project.
  • reset.sass — resets everything to zero margin, padding + some black magic hacks to fix browsers inconsistencies
  • spaces.sass — use to modify the default spacing between objects
  • type.sass — stores all typography related style definitions

Layouts

  • *.sass — place for general layout styling. Store body, wrapper, content and other layout specific blocks here. Add a link to a config.sass to use shared mixins with variables. You can store all kinds of layouts here like: application.sass, homepage.sass, admin.sass

Library

  • *.sass — place for groups like buttons, text-inputs, selects, radios, checkboxes and other elements, which you can combine. Add a link to a config.sass to use shared mixins with variables. To specify multiple libraries for different layouts move groups to separate folders inside the library: /library/application/ /library/admin/

Modules

  • *.sass — use to create modules such as login-block.sass, modal.sass, votebar.sass, dropdown.sass, lightbox.sass and so on. Add a link to a config.sass to use shared mixins with variables.
  • To specify multiple modules sets for different layouts create folders inside:

    /modules/application/

    /modules/admin/

  • If module gets too big, slice it to keep code small and flat structure:

    /modules/player-info.sass

    /modules/player-status.sass

    /modules/player-controls.sass

    /modules/player-states.sass
